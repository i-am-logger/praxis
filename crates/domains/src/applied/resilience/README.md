# Resilience — recovery patterns, backoff, supervision

Companion to [Dependability](../dependability/) (#122). Where Dependability defines what errors ARE, Resilience defines how systems RESPOND: stability patterns that prevent cascading failure, backoff strategies that de-correlate retries, supervision trees that isolate crash blast-radius, and recovery-oriented computing patterns that restore correct service.

Per user direction surfacing PR #120 review:

> we need ontology understanding of erroring and recovery, backoff, etc

## Functors and Endofunctors

**Deferred** — two follow-ups:

1. **`Resilience → Dependability`** — each pattern targets a specific fault class (circuit-breaker ↦ cascade failures; backoff ↦ transient faults; supervision ↦ OperationalFaults). Strict-Functor laws likely fail at this scale for the same reason the Dependability→Diagnostics functor was removed: dense-to-kinded mapping collapses morphisms many-to-one, breaking `F(g ∘ f) = F(g) ∘ F(f)`. Needs lax-functor framework support.
2. **Endofunctor instances** — the CircuitBreaker state machine is a natural endofunctor candidate, but only on a *single-kinded* version of the transition edges (currently TripsTo, CoolsTo, ResolvesTo, RelapsesTo are distinct kinds, so a rotation endofunctor can't preserve kinds). Exponential backoff is an endofunctor on Duration — needs Duration as a first-class object of a time category. Both tracked as #124 follow-ups.
